
## S3

概要

- Amazon Simple Storage Service (S3) はオブジェクトストレージサービス。高い耐久性（99.999999999%）と可用性を提供し、静的ウェブホスティング、バックアップ、データレイク、ログ保存など幅広い用途で使われます。

## 重要トピック（詳細）

### ストレージクラス

- `S3 Standard`:
  - 頻繁にアクセスされるデータ向けの標準クラス。
  - 複数 AZ に保存され、高い可用性と低レイテンシを提供。
  - 静的 Web サイト、アプリケーションデータ、頻繁に参照される画像や動画に適する。
- `S3 Intelligent-Tiering`:
  - アクセス頻度が読めないデータ向け。
  - オブジェクトのアクセス状況に応じて、自動的に低コスト階層へ移動してコストを最適化する。
  - アクセスパターンが変動するログ、分析データ、ユーザー生成コンテンツに向く。
  - 監視と自動階層化のための少額のモニタリング料金が発生する。
- `S3 Standard-IA`:
  - 低頻度アクセスだが、必要時には即時取得したいデータ向け。
  - 複数 AZ に保存されるため耐障害性を確保しつつ、`Standard` よりストレージ単価が低い。
  - バックアップ、DR データ、長めに保管するが時々読むファイルに適する。
  - 取得料金が発生するため、頻繁アクセスには不向き。
- `S3 One Zone-IA`:
  - `Standard-IA` と同様に低頻度アクセス向けだが、単一 AZ のみで保存する。
  - 再作成可能なバックアップや二次コピーなど、AZ 障害時に失っても許容できるデータ向け。
  - コストは低いが、AZ 障害への耐性は `Standard-IA` より弱い。
- `S3 Glacier Instant Retrieval`:
  - アーカイブ用途だが、ミリ秒単位で即時取得したいデータ向け。
  - 医療画像、メディア資産、長期保存が必要だが即時参照の可能性があるデータに向く。
  - アーカイブ系の中では取得性が高い一方、取得コストは考慮が必要。
- `S3 Glacier Flexible Retrieval`:
  - 取り出し時間とコストのバランスを取るアーカイブクラス。
  - Expedited、Standard、Bulk など複数の取得オプションがあり、数分から数時間で取り出せる。
  - バックアップ、災害対策データ、定期監査用データに適する。
- `S3 Glacier Deep Archive`:
  - 最も低コストな長期アーカイブ向けクラス。
  - 取り出しに長時間かかるため、法令保存、監査証跡、長期保管ログなど、めったに参照しないデータに向く。
  - 「最安だが取り出しが遅い」ことが最大の特徴。
- `S3 Express One Zone`:
  - 単一 AZ で非常に低レイテンシかつ高いリクエスト性能が必要なワークロード向け。
  - 頻繁アクセスされる一時データや高性能分析処理などに使う。
  - 単一 AZ 保管のため、マルチ AZ 耐障害性が必要なデータには不向き。

### ストレージクラス選定の目安

- 頻繁アクセス: `S3 Standard`
- アクセス頻度が読みにくい: `S3 Intelligent-Tiering`
- 低頻度アクセスだが即時取得したい: `S3 Standard-IA`
- 低頻度アクセスかつ単一 AZ でよい: `S3 One Zone-IA`
- アーカイブだが即時取得したい: `S3 Glacier Instant Retrieval`
- アーカイブで数分から数時間の復元を許容: `S3 Glacier Flexible Retrieval`
- ほぼ参照しない超長期保管: `S3 Glacier Deep Archive`

### 試験での見分け方

- 「頻繁に使う」なら `Standard`。
- 「アクセス頻度が不明」なら `Intelligent-Tiering`。
- 「即時取得が必要な低頻度アクセス」なら `Standard-IA`。
- 「AZ 障害時に失ってもよい二次データ」なら `One Zone-IA`。
- 「アーカイブだがすぐ読みたい」なら `Glacier Instant Retrieval`。
- 「最安で長期保存、取り出しは遅くてよい」なら `Glacier Deep Archive`。

### バージョニングとライフサイクル

- バージョニングを有効化するとオブジェクトの変更や削除の履歴を保持し、誤削除から復旧可能。
- ライフサイクルポリシーでオブジェクトを別クラスへ移行したり、期限で削除したりできる。

### アクセス制御とセキュリティ

- バケットポリシー、IAM ポリシー、ACL、S3 Access Points によるアクセス制御。
- SSE-S3 (S3 管理のキー), SSE-KMS (KMS 管理キー), SSE-C (顧客提供キー) の暗号化オプション。
- S3 Block Public Access: 誤って公開されるリスクを防ぐための設定。

### レプリケーションと整合性

- Cross-Region Replication (CRR): バケットのオブジェクトを他リージョンに非同期で複製して DR を実現。
- Same-Region Replication (SRR): 同一リージョン内での複製。
- S3 の読み取り整合性: PUT 後の強い整合性が保証される（オブジェクト取得は強い整合性）。

### パフォーマンスと設計

- パーティショニングキー（プレフィックス）設計は過去の注意点だったが、現在の S3 はスケーラビリティが改善されている。
- 大規模並列アップロードはマルチパートアップロードを利用。

### イベント通知と統合

- S3 イベント通知で Lambda, SQS, SNS に通知を飛ばし、イベント駆動処理を構築可能。

## CLI / IaC の例

```bash
# バケット作成
aws s3api create-bucket --bucket my-bucket --region ap-northeast-1 --create-bucket-configuration LocationConstraint=ap-northeast-1

# バージョニング有効化
aws s3api put-bucket-versioning --bucket my-bucket --versioning-configuration Status=Enabled

# オブジェクトを Glacier Deep Archive に移行するライフサイクルの例は CloudFormation / CLI で設定
```

## コスト面の考慮

- ストレージ容量だけでなく、リクエスト数、データ取り出し（取り出しリクエスト + データ転送）、レプリケーションやライフサイクル移行のコストも考慮。

## 試験向けポイント / よく出るトピック

- S3 は 11 9's（99.999999999%）の耐久性を提供。
- バージョニング + ライフサイクル + レプリケーションの組み合わせによるデータ保護戦略。
- S3 エンドポイントを使うことでプライベートネットワークからアクセス可能になり、NAT や IGW を経由しない。
