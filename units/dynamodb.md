# DynamoDB

概要

- Amazon DynamoDB はフルマネージドな NoSQL キーバリューデータベースで、単一桁ミリ秒の低レイテンシを大規模に提供します。

## 主要トピック

### キャパシティモード

- プロビジョンド: 読み書き容量（RCU/WCU）を明示的に設定。ホットキーに注意してパーティション設計。
- オンデマンド: 使用量に応じて自動スケール、アクセスパターンが予測しにくい場合に便利。

### テーブル設計（アクセスパターン駆動）

- パーティションキー（ハッシュ）とソートキーを使った単一テーブル設計が一般的。
- 逆正規化、セカンダリインデックス（GSI/LSI）や複合キー設計で多様なクエリに対応。
- ユニークなアクセスパターンごとに設計を行う（クエリ数が多い属性は GSI を検討）。

### インデックス

- GSI（グローバルセカンダリインデックス）: テーブルとは別のパーティションキーでクエリ可能。書き込み時に追加コスト。
- LSI（ローカルセカンダリインデックス）: 同一パーティションキーで別のソートキーを使用、テーブル作成時にのみ定義可能。

### 一貫性モデルとトランザクション

- 読み取りは最終整合性がデフォルト。強い整合性を要求する場合はオプションで指定可能（追加コスト）。
- トランザクション（TransactWrite/TransactGet）で複数アイテムの原子的な更新をサポート。

### ストリームとイベント駆動

- DynamoDB Streams を使ってテーブルの変更をキャプチャし、Lambda や Kinesis で処理可能。

### キャッシュと高速化

- DAX（DynamoDB Accelerator）: インメモリキャッシュで読み取りレイテンシをマイクロ秒に短縮（キャッシュ一貫性の考慮が必要）。

### バックアップとレプリケーション

- On-demand スナップショットと Point-in-Time Recovery（PITR）を利用して復旧戦略を設計。
- Global Tables を使って複数リージョンにマルチマスター構成を構築し、DR とリージョン近接性を向上。

### パフォーマンスとコストの考慮

- ホットパーティションを避けるためキー設計を工夫する。
- 読み取りタイプ（強い整合性/最終整合性/トランザクション）によるコスト差を把握。

## CLI / IaC の例

```bash
# テーブル作成（オンデマンド）
aws dynamodb create-table \
  --table-name Orders \
  --attribute-definitions AttributeName=PK,AttributeType=S AttributeName=SK,AttributeType=S \
  --key-schema AttributeName=PK,KeyType=HASH AttributeName=SK,KeyType=RANGE \
  --billing-mode PAY_PER_REQUEST

# GSI の追加（CloudFormation/CDK 推奨で変更管理）
```

## 試験向けポイント / よく出るトピック

- テーブル設計はアクセスパターンに基づく（正規化より逆正規化が多い）。
- オンデマンドは運用負荷が少ないが、プロビジョンドは長期負荷でコスト優位になることがある。
- Global Tables はマルチリージョンの読み取り/書き込みをサポートするが、コンフリクトやレプリケーション遅延の考慮が必要。

## サンプル問題

1. DynamoDB のホットパーティションとは何か。防止するための設計戦略を 3 つ挙げよ。

2. グローバルテーブルを利用するメリットと注意点を説明してください。
