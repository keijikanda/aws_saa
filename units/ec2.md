# EC2

概要

- Amazon Elastic Compute Cloud (EC2) はスケーラブルな仮想サーバを提供するサービス。インスタンスはさまざまな構成（CPU、メモリ、ストレージ、ネットワーク）で提供され、オンデマンド、リザーブド、スポット等の課金モデルがあります。

## 重要ポイント（詳細）

### インスタンスタイプとユースケース

- 汎用 (t, m シリーズ): バランスの取れた CPU / メモリ。開発環境や汎用アプリ。
- コンピューティング最適化 (c シリーズ): CPU バウンドなワークロード（バッチ処理、ハイパフォーマンス計算の一部）。
- メモリ最適化 (r, x シリーズ): データベースやインメモリキャッシュ（Redis など）。
- ストレージ最適化 (i, d シリーズ): ローカル NVMe を利用する I/O 集約ワークロード。
- GPU / 機械学習 (p, g シリーズ): 機械学習やグラフィックス処理。
- バースト可能 (t シリーズ): 平常時負荷が低く、必要時にバーストするワークロード向け。

### AMI とストレージ

- AMI: インスタンスのイメージ（OS、アプリケーション、起動設定を含められる）。カスタム AMI を作成して標準化可能。
- EBS (Elastic Block Store): 永続ブロックストレージ。種類: gp3/gp2（汎用 SSD）、io1/io2（高 IOPS）、st1/sc1（スループット最適化）
- EBS スナップショット: 増分バックアップで S3 に保存される（管理は AWS が行う）。
- インスタンスストア: ローカルの一時ストレージ（再起動や停止でデータが消える）。

### ネットワーク

- パブリック IP とプライベート IP。Elastic IP は固定パブリック IPv4 を提供。
- ENI (Elastic Network Interface): 複数 ENI を付与してネットワーク分離やフェイルオーバーを実現。
- セキュリティグループ: ステートフルな仮想ファイアウォール（インバウンド/アウトバウンドルール）。
- ネットワーク ACL (NACL): ステートレスでサブネット単位に適用。セキュリティグループと併用。

### IAM と認証

- EC2 に IAM ロールをアタッチして、インスタンス上のアプリが AWS リソースへ安全にアクセスできるようにする（長期的なアクセスキーをインスタンスに置かない）。
- SSH はキーペアを使用した公開鍵認証。EC2 Instance Connect や Systems Manager Session Manager を使えばパブリック IP を開けずに接続可能。

### 起動オプションと運用

- 起動テンプレート / 起動設定: ASG（Auto Scaling Group）との統合のための再利用可能な構成。
- ユーザーデータ (user-data): インスタンス初回起動時に実行されるスクリプト。CloudInit と組み合わせて設定。
- 停止と終了の違い: 停止後は EBS ルートボリュームを保持できるが、終了すると削除設定によってはボリュームが削除される。

### スポット・リザーブド・オンデマンド

- オンデマンド: 柔軟で即時利用可能、コストは高め。
- リザーブドインスタンス（RI）/Savings Plans: 1 年/3 年のコミットで割引が得られる。長期稼働のワークロードに有利。
- スポットインスタンス: 余剰キャパシティを安価に使えるが中断（終了/再割り当て）のリスクがある。可用性の低いバッチやワークロード、スポットフリートで補完。

### 可用性とスケーラビリティ

- 複数 AZ に分散して配置することで単一 AZ 障害を回避。
- Auto Scaling とロードバランサ（ALB/NLB）を用いて自動スケールとヘルスチェックで耐障害性を確保。

### モニタリングとトラブルシュート

- CloudWatch Metrics: CPUUtilization, NetworkIn/Out, DiskReadOps/WriteOps, StatusCheckFailed などを監視。
- CloudWatch Logs にアプリログを送る。CloudWatch Alarms で自動化（例: スケーリングのトリガー）。
- システムステータスチェック（ハードウェア）とインスタンスステータスチェック（OS）を理解しておく。

## CLI / IaC の例

```bash
# EC2 インスタンスの起動 (例)
aws ec2 run-instances \
	--image-id ami-0123456789abcdef0 \
	--count 1 \
	--instance-type t3.micro \
	--key-name my-key \
	--subnet-id subnet-01234567 \
	--security-group-ids sg-01234567

# EBS スナップショット作成
aws ec2 create-snapshot --volume-id vol-01234567 --description "daily-backup"
```

## 試験向けポイント / よく出るトピック

- セキュリティグループはステートフル（レスポンスは自動で許可される）。NACL はステートレスで双方向ルールが必要。
- スポットインスタンスの中断通知を使ったフェイルオーバー手法（終了前に再配置や保存処理を行う）。
- ENI と Elastic IP の違い（ENI はネットワークインターフェイス、Elastic IP は静的パブリック IP）。

## 模擬問題（選択肢付き）

1) 可用性を最優先する Web サービスを EC2 で設計する際、最も適切な構成はどれか？

A. 単一 AZ に大きな EC2 インスタンスを置き、EBS をアタッチする。
B. 複数 AZ にまたがるサブネットに EC2 を配置し、ALB を使ってトラフィックを分散、ASG でインスタンスを自動復旧させる。
C. すべてのトラフィックを単一のパブリック IP に集める。
D. スポットインスタンスのみを使い、単一 AZ 内でスケールする。

2) EBS スナップショットに関して正しいのはどれか？

A. スナップショットは常にフルバックアップである。
B. スナップショットは増分バックアップで、効率的にストレージを使用する。
C. スナップショットからはルートボリュームは復元できない。
D. スナップショットは自動的に暗号化されない。

3) セキュリティグループと NACL の違いで正しいものはどれか？

A. セキュリティグループはステートレスで、NACL はステートフルである。
B. セキュリティグループはインスタンスレベルのステートフル制御で、NACL はサブネットレベルのステートレス制御である。
C. NACL はインスタンスに直接アタッチされる。
D. 両方とも自動でレスポンスを許可する。

4) スポットインスタンスを利用する際の主なリスクと対策の組合せで正しいのはどれか？

A. リスク: 中断される。対策: スポットのみで重要なサービスを運用する。
B. リスク: 中断される。対策: スポットフリートやスポット + オンデマンド、チェックポイントや Auto Scaling を組み合わせる。
C. リスク: 常に高コスト。対策: RI に切り替える。
D. リスク: 永続ストレージが失われる。対策: インスタンスストアを使用する。

5) EC2 インスタンスから S3 に安全にアクセスさせる一般的な方法として最も適切なのはどれか？

A. インスタンスに長期的なアクセストークン（アクセスキー）を埋め込む。
B. IAM ロールを EC2 にアタッチし、必要な権限を付与する。
C. S3 バケットをパブリックにしてアクセスを許可する。
D. SSH トンネル経由で接続させる。

---

回答と簡単な解説：
1) B（マルチ AZ + ALB + ASG が高可用性の設計）
2) B（EBS スナップショットは増分バックアップ）
3) B（セキュリティグループはステートフル、NACL はステートレス）
4) B（中断リスクを考慮し、柔軟な設計で補う）
5) B（IAM ロールが安全で推奨される方法）
